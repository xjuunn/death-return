# AGENTS.md

本文件定义 `death-return` 仓库内协作者、自动化代理和维护者的统一工作约束。

目标不是增加流程成本，而是确保文档、代码和提交结果具备稳定质量，并且适合持续维护。

## 项目定位

`death-return` 是一个面向 Fabric 的死亡坐标记录模组，重点在于：

- 功能范围明确
- 交互简洁直接
- 服务端逻辑优先
- 命令和提示具备可读性
- 配置项保持克制

新改动应优先服务这些目标，不应无节制扩大模组职责。

## 工作原则

- 保持功能极简，只解决与死亡记录、查看、传送和管理相关的问题。
- 优先使用稳定、直接、可维护的实现，不为短期便利引入复杂抽象。
- 任何新增能力都应先考虑是否适合现有命令体系和配置结构。
- 文案应使用中文，表达准确、克制、面向实际使用者。
- 禁止在文档、提示文本、提交说明中使用 emoji。

## 代码约束

- 保持现有 Kotlin 代码风格，不引入无必要的风格混杂。
- 优先使用 Mojang mappings 对应 API，不要混用 Yarn 命名。
- 源码文件默认使用 ASCII 可安全保存的内容；中文文本优先采用不会引发编码问题的写法。
- 新增逻辑应尽量放在已有职责清晰的模块中，不随意堆叠到入口类。
- 注释只说明必要背景、兼容逻辑或非直观点，不写重复代码含义的注释。

## 文档要求

- `README.md` 必须适合 GitHub 原生渲染。
- 标题层级应清晰，避免大段无结构说明。
- 优先使用表格、短段落和扁平列表组织信息。
- 安装、命令、配置、兼容版本、数据存储和构建方式应保持同步更新。
- 文档内容应与实际代码行为一致，不允许“文档先行但代码未实现”的状态长期存在。

## 配置与命令

- 新配置项必须有明确用途，且默认值应面向大多数服务器场景。
- 命令命名应保持简短、可预测，不引入难记的别名体系。
- 任何新增命令都必须考虑权限边界、失败提示和用户可理解性。
- 聊天交互应兼顾美观与实用，优先清晰，不追求花哨。

## 提交规范

- 提交粒度应尽量单一，一个 commit 只解决一类问题。
- 推荐使用以下前缀：
  - `feat:` 新功能
  - `fix:` 缺陷修复
  - `refactor:` 结构调整
  - `docs:` 文档更新
  - `chore:` 杂项维护
- 不要把无关修改混入同一个提交。

## 验证要求

在环境允许的前提下，改动后应至少执行与任务相关的验证：

- 代码变更：`compileKotlin` 或 `build`
- 启动链路变更：`runClient` 或 `runServer`
- 文档变更：检查 Markdown 在 GitHub 风格下的可读性

如果由于本机环境限制无法完成验证，必须在最终说明中明确指出原因。

## 禁止事项

- 不要为了“更灵活”引入没有实际收益的复杂配置。
- 不要添加与项目目标无关的 UI、菜单或客户端渲染逻辑。
- 不要使用破坏工作区的命令覆盖用户已有修改。
- 不要在文档中使用营销式、夸张式或口号化表达。

## 文档同步

当以下内容发生变化时，必须同步更新 `README.md`：

- 命令
- 配置项
- 兼容版本
- 安装方式
- 存储格式或路径

如果仓库未来引入测试、发布流程或多平台分发，也应继续在本文件补充相应规范。
